Use `<RevealOnHover>` to hide secondary actions at rest and reveal them when the user hovers or focuses within the container. Actions are always visible on touch devices.

## Callback Children (Custom Elements)

When you need to apply reveal behavior to a non-Flex element (e.g., a grid row or table cell), pass a callback child. The callback receives `{className}` which you apply to your own element.

```jsx
<RevealOnHover>
  {({className}) => (
    <tr className={className}>
      <td>Tag name</td>
      <td>
        Tag value
        <RevealOnHover.Action>
          <Button
            size="zero"
            variant="transparent"
            aria-label="Copy"
            icon={<IconCopy size="xs" />}
          />
        </RevealOnHover.Action>
      </td>
    </tr>
  )}
</RevealOnHover>
```

This is useful for grid layouts, table rows, or any case where wrapping in a `Flex` would break the parent layout.

## How It Works

- Actions use `opacity: 0` + `pointer-events: none` at rest, ensuring space is reserved (no layout shift) while preventing ghost clicks.
- On `:hover` or `:focus-within`, actions transition to `opacity: 1` + `pointer-events: auto`.
- On touch devices (`@media (hover: none)`), actions are always visible since there is no hover interaction.
- Transitions use `motion.enter.moderate` (160ms) to appear and `motion.exit.fast` (120ms) to disappear.

## Accessibility

- **Keyboard**: Actions remain focusable at all times. When a user tabs to a hidden action, `:focus-within` fires on the container and reveals all actions.
- **Screen readers**: Actions are in the DOM with full opacity semantics — screen readers can discover them regardless of visual state.
- **Touch devices**: Actions are persistently visible since hover is not available.
- **WCAG 1.4.13**: Content revealed on hover is hoverable and persistent.
- **WCAG 3.2.7**: Use this only for secondary/supplementary actions. Primary actions should always be visible.
